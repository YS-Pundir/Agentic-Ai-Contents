## Question 1 *(Easy | MCQ — Single Correct)*

You are helping an e-commerce company move from a demo RAG system to a real customer support assistant. In the demo, policy details were stored as Python strings inside a list. The company now stores return, refund, and shipping policies as PDF files.

Which step should be added first so the RAG system can use these real policy files?

- A) Hardcode each PDF page manually inside a Python list
- B) Load the PDF documents into the Python program using a PDF-reading library
- C) Ask the LLM to guess the policy content without reading the files
- D) Store only the PDF file names in ChromaDB

**Correct Answer:** B) Load the PDF documents into the Python program using a PDF-reading library

**Answer Explanation:**
The first real-world step is document ingestion: loading external documents into the program. The session specifically uses `pypdf` and `PdfReader` to read PDF pages.
- A is incorrect — it repeats the toy approach and does not scale.
- C is incorrect — it would create ungrounded answers because the LLM has no policy context.
- D is incorrect — file names alone do not contain the actual policy text needed for retrieval and generation.

## Question 2 *(Easy | MCQ — Single Correct)*

A PDF extraction function returns text with repeated blank lines, page headers, footers, and irregular spacing. Before chunking this text, what should the pipeline do?

- A) Clean the extracted text by removing redundant whitespace and noisy elements
- B) Generate embeddings immediately without modifying the text
- C) Store the raw PDF binary directly in the vector database
- D) Increase the LLM temperature so it can handle messy text creatively

**Correct Answer:** A) Clean the extracted text by removing redundant whitespace and noisy elements

**Answer Explanation:**
Cleaning is needed because raw PDF text often contains noise such as headers, footers, extra blank lines, and random whitespace. The session shows using operations such as `.replace()`, `.strip()`, and regular expressions to make the text usable.
- B is incorrect — noise would enter downstream retrieval.
- C is incorrect — raw binary does not support text-based similarity search.
- D is incorrect — temperature affects answer creativity, not document cleaning.

## Question 3 *(Easy | MCQ — Single Correct)*

In the session's RAG pipeline, why are large policy documents split into smaller chunks?

- A) To make each document look shorter in the file explorer
- B) To avoid context-window overload and make retrieval more precise
- C) To remove the need for embeddings completely
- D) To make the LLM ignore metadata

**Correct Answer:** B) To avoid context-window overload and make retrieval more precise

**Answer Explanation:**
Chunking prevents the system from sending huge documents to the LLM for every query. Smaller chunks reduce cost, fit better within context-window limits, and make similarity search more targeted.
- A is incorrect — unrelated to the RAG pipeline.
- C is incorrect — embeddings are still needed for similarity search.
- D is incorrect — metadata remains useful for filtering and traceability.

## Question 4 *(Easy | MCQ — Single Correct)*

A developer creates chunks of 120 words from policy pages and uses an overlap of 30 words. What is the main purpose of this overlap?

- A) To preserve context across chunk boundaries
- B) To make every chunk exactly twice as long
- C) To delete duplicate information from the source PDF
- D) To avoid using a vector database

**Correct Answer:** A) To preserve context across chunk boundaries

**Answer Explanation:**
Chunk overlap repeats the last few words of one chunk at the beginning of the next chunk so meaning is not lost when a sentence or idea crosses a boundary.
- B is incorrect — overlap does not double every chunk length.
- C is incorrect — overlap intentionally duplicates a small amount of text; it does not remove duplicates from the PDF.
- D is incorrect — unrelated; a vector database is still used.

## Question 5 *(Moderate | MCQ — Single Correct)*

You are designing the ChromaDB records for an e-commerce policy assistant. For each chunk, you want users to receive accurate answers and you also want the team to trace where an answer came from.

Which combination should be stored for each chunk?

- A) Only the original PDF file name
- B) Chunk text, embedding vector, and metadata such as source file, page number, and policy type
- C) Only the final answer generated by the LLM
- D) Only the OpenAI API key and collection name

**Correct Answer:** B) Chunk text, embedding vector, and metadata such as source file, page number, and policy type

The session explains that each chunk should store the chunk text, its embedding vector, and metadata. The text is needed as retrieved context, the vector is needed for similarity search, and metadata helps with filtering and source tracing.
- A is incorrect — it lacks the content and vector representation.
- C is incorrect — outputs are stored instead of searchable knowledge.

## Question 6 *(Moderate | MCQ — Single Correct)*

A user asks, *"How long does a refund take?"* The system converts this query into an embedding and searches ChromaDB for the top 3 most similar chunks before building the final prompt.

Which part of RAG is mainly responsible for this search step?

- A) Generator
- B) Retriever
- C) OCR engine
- D) Virtual environment

**Correct Answer:** B) Retriever

**Answer Explanation:**
The **Retriever** converts the query into an embedding and performs similarity search in the vector database to return the most relevant chunks. The Generator uses those chunks later to create a natural-language answer.
- A is incorrect — the Generator consumes retrieved chunks but does not run the similarity search.
- C is incorrect — OCR is only relevant for image-based or scanned text extraction.
- D is incorrect — a virtual environment is a project setup tool, not a RAG component.

## Question 9 *(Hard | MSQ — Multiple Correct)*

Your RAG assistant answers refund questions correctly, but the company updates `refund_policy.pdf`. The old chunks in ChromaDB still mention the previous refund timeline.

Which actions are necessary to keep the knowledge base current? *(Select all that apply.)*

- A) Delete chunks related to the old refund policy from ChromaDB
- B) Re-load and clean the updated refund PDF
- C) Re-chunk, re-embed, and store the updated chunks
- D) Keep both old and new refund chunks so the LLM can choose whichever sounds better

**Correct Answers:** A, B, C

**Answer Explanation:**
- **A is correct** — Stale chunks can surface outdated policy unless removed or superseded deliberately.
- **B is correct** — The updated source must be ingested and cleaned like any other indexing run.
- **C is correct** — New chunks need fresh embeddings and storage so retrieval reflects the latest text.
- **D is incorrect** — Keeping outdated chunks invites contradictory retrieval and wrong final answers.

## Question 10 *(Hard | MSQ — Multiple Correct)*

You are reviewing this prompt template for the Generator:

```text
Answer the customer's question using ONLY the policy information provided below.
Do NOT make up any details. If the answer is not in the context, say: "I don't have that information."
Keep your answer simple, clear, and customer-friendly.
```

Which benefits does this prompt design provide in the RAG pipeline? *(Select all that apply.)*

- A) It reduces hallucination by forcing the model to use only retrieved context
- B) It gives the model a fallback when the retrieved chunks do not contain the answer
- C) It guarantees that similarity search will always retrieve the correct chunks
- D) It controls the tone of the final answer for a customer support scenario

**Correct Answers:** A, B, D

**Answer Explanation:**
- **A is correct** — Instructions to use only retrieved policy context ground the Generator and curb hallucinations.
- **B is correct** — The fallback sentence guides behavior when context is insufficient.
- **D is correct** — "Simple, clear, and customer-friendly" shapes tone for support use cases.
- **C is incorrect** — Retrieval quality depends on embeddings, chunking, metadata, and search; prompts cannot guarantee perfect retrieval.
